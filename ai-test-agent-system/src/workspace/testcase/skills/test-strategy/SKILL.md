# 测试策略制定 Skill

## 激活场景
- 需求分析完成后，开始制定整体测试方案
- 用户询问"应该测哪些方面？"
- 用户需要评估测试覆盖是否充分
- 项目有时间约束，需要优先级排序

## 测试类型选择决策树

### 🔹 功能测试（必选）
**适用条件**：所有功能性需求
```
正向流程测试   → 验证核心业务路径（Happy Path）
逆向流程测试   → 验证错误处理与异常场景
边界条件测试   → 验证输入边界处理
等价类测试     → 验证数据分类处理
```

### 🔹 接口测试（API项目必选）
**适用条件**：存在HTTP API、RPC、消息队列
```
契约验证       → 请求/响应格式符合接口文档
状态码验证     → HTTP状态码语义正确
认证鉴权       → Token/Session/OAuth验证
幂等性验证     → 重复请求处理
```

### 🔹 安全测试（涉及用户数据必选）
**适用条件**：认证、授权、用户数据、支付
```
SQL注入测试    → 数据库查询注入防护
XSS测试        → 跨站脚本防护
CSRF测试       → 跨站请求伪造防护
越权测试       → 水平/垂直越权验证
敏感数据保护   → 密码、身份证、手机号脱敏
```

### 🔹 性能测试（高并发场景必选）
**适用条件**：用户量>1000、高频接口、数据密集型操作
```
基准性能测试   → 单用户响应时间基线
负载测试       → 预期并发用户数下的表现
压力测试       → 极限并发用户数
稳定性测试     → 7×24小时长时间运行
```

### 🔹 兼容性测试（多端产品选择）
**适用条件**：Web端、移动端、多浏览器
```
浏览器兼容     → Chrome/Firefox/Safari/Edge
设备兼容       → iOS/Android 多机型
分辨率适配     → 375px/768px/1024px/1920px
OS版本兼容     → iOS 14+/Android 8+
```

### 🔹 可用性测试（ToC产品选择）
**适用条件**：面向终端用户的消费级产品
```
交互逻辑验证   → 操作反馈是否即时
错误提示质量   → 错误信息是否清晰可操作
无障碍验证     → WCAG 2.1 AA级标准
```

## 优先级策略矩阵

基于**业务影响 × 风险概率**进行优先级评定：

```
              | 低概率 | 高概率
高业务影响    |   P1   |   P0
低业务影响    |   P3   |   P2
```

### 优先级定义（强制遵循）
| 级别 | 名称 | 执行时机 | 通过率要求 | 典型场景 |
|------|------|---------|-----------|---------|
| **P0** | 阻塞级 | 每次构建/发布前 | 100% | 核心支付流程、用户登录 |
| **P1** | 高优先级 | 每个Sprint结束 | ≥95% | 主要业务功能、数据准确性 |
| **P2** | 中优先级 | 版本发布前 | ≥90% | 辅助功能、优化项 |
| **P3** | 低优先级 | 有时间则执行 | 尽力覆盖 | 边缘场景、UI细节 |

## 特殊策略场景

### 时间紧迫时的策略（冒烟测试套件）
> 当发布时间压缩时，快速构建最小有效测试集：
> 1. 只执行P0级用例（核心流程不超过20个用例）
> 2. 覆盖所有主要功能的正向流程
> 3. 覆盖最高风险的异常场景（支付失败、数据丢失）

### 新功能 vs 存量功能策略
- **新功能**：全面覆盖，包含边界和异常（设计深度=深）
- **存量功能**（受新功能影响）：聚焦影响区域（设计深度=中）
- **存量功能**（未受影响）：仅冒烟验证（设计深度=浅）

## 测试类型选择的量化评分卡

企业级测试策略不能仅凭经验判断，需要对每个测试类型进行量化评估。

### 评分维度与权重

| 评分维度 | 权重 | 评估标准 |
|---------|------|---------|
| 业务风险 | 30% | 该功能失败的业务影响程度（1-5分） |
| 技术风险 | 25% | 技术复杂度、变更频率、历史缺陷密度（1-5分） |
| 合规要求 | 20% | 行业法规是否强制要求（1-5分） |
| 用户影响 | 15% | 影响用户数量和使用频率（1-5分） |
| 成本效益 | 10% | 测试投入与预期收益的匹配度（1-5分） |

### 测试类型执行决策

对每个测试类型，计算加权总分：

```
总分 = 业务风险×0.3 + 技术风险×0.25 + 合规要求×0.2 + 用户影响×0.15 + 成本效益×0.1
```

| 总分范围 | 决策 | 说明 |
|---------|------|------|
| 4.0-5.0 | **必须执行** | 核心测试类型，投入最大资源 |
| 3.0-3.9 | **建议执行** | 重要测试类型，标准资源投入 |
| 2.0-2.9 | **有条件执行** | 视时间和资源情况决定 |
| 1.0-1.9 | **可选执行** | 仅在有富余资源时考虑 |
| <1.0 | **不执行** | 当前版本暂不覆盖 |

### 评分示例

**场景**：电商订单支付模块

| 测试类型 | 业务风险 | 技术风险 | 合规要求 | 用户影响 | 成本效益 | 加权总分 | 决策 |
|---------|---------|---------|---------|---------|---------|---------|------|
| 功能测试 | 5 | 4 | 3 | 5 | 5 | 4.45 | 必须执行 |
| 接口测试 | 4 | 5 | 3 | 3 | 4 | 3.95 | 建议执行 |
| 安全测试 | 5 | 4 | 5 | 4 | 3 | 4.30 | 必须执行 |
| 性能测试 | 3 | 3 | 2 | 4 | 3 | 3.00 | 有条件执行 |
| 兼容测试 | 2 | 2 | 1 | 3 | 2 | 1.95 | 可选执行 |

## 测试环境策略

### 环境拓扑规划

```
[开发环境] → [测试环境] → [预发布环境] → [生产环境]
     ↑            ↑              ↑
   单元测试    集成测试      验收测试
   接口测试    系统测试      回归测试
   冒烟测试    性能测试
               安全测试
```

### 环境配置要求

| 环境 | 用途 | 数据要求 | 配置要求 |
|------|------|---------|---------|
| 测试环境 | 功能/接口/安全测试 | 合成测试数据（不可使用生产数据） | 与生产环境架构一致，资源配置可降级 |
| 性能环境 | 性能/压力/稳定性测试 | 与生产等比例数据量 | 与生产环境资源配置一致或按比例缩放 |
| 预发布环境 | 回归测试/UAT验收 | 生产数据脱敏副本 | 与生产环境完全一致 |

### 数据准备策略

| 数据类型 | 来源 | 注意事项 |
|---------|------|---------|
| 基础数据 | 手工构造 + 脚本生成 | 覆盖所有业务场景所需的基准数据 |
| 边界数据 | 基于字段约束构造 | 覆盖 min-1, min, max, max+1 |
| 性能数据 | 批量生成脚本 | 数据量需达到性能测试目标量级 |
| 安全数据 | 标准Payload库 + 定制 | 确保Payload符合实际攻击场景 |
| **严禁使用生产数据** | - | 违反个保法/GDPR，必须使用脱敏或合成数据 |

### Mock 策略

| 依赖类型 | Mock方式 | 适用场景 |
|---------|---------|---------|
| 第三方支付 | Mock Server | 支付回调、退款通知 |
| 短信/邮件服务 | 本地Stub | 验证码发送、通知推送 |
| 外部API | WireMock/MockServer | 身份认证、数据同步 |
| 消息队列 | 内存队列 | 异步任务、事件驱动 |

## 自动化测试策略

### 自动化适用性评估

对每个功能模块，评估是否适合自动化：

| 评估维度 | 适合自动化 | 不适合自动化 |
|---------|-----------|------------|
| 执行频率 | 每次构建都执行 | 偶尔执行 |
| 稳定性 | 功能稳定，UI/接口不常变 | 频繁变更，尚未稳定 |
| 复杂度 | 步骤明确，判定标准清晰 | 依赖主观判断、探索性测试 |
| 执行成本 | 手工执行耗时 > 自动化维护成本 | 手工执行耗时 < 自动化维护成本 |
| 回归价值 | 高价值回归场景 | 一次性验证场景 |

### 自动化分层策略（测试金字塔）

```
         /\
        /  \     E2E 端到端测试（10%）
       /    \    → 核心业务流程验证
      /------\
     /        \   接口/服务测试（30%）
    /          \  → API契约、业务逻辑验证
   /------------\
  /              \ 单元测试（60%）
 /                \ → 函数/方法级验证
/__________________\
```

**企业级目标**：
- 单元测试覆盖率 ≥ 80%
- 接口测试覆盖所有外部接口
- E2E 测试覆盖所有 P0 业务流程

### 自动化与手工用例的对应关系

| 手工用例优先级 | 自动化建议 | 说明 |
|---------------|-----------|------|
| P0 | **必须自动化** | 每次构建自动执行，作为发布门禁 |
| P1 | **建议自动化** | Sprint 结束前的回归执行 |
| P2 | **有条件自动化** | 功能稳定后补充自动化 |
| P3 | **不自动化** | 保留为手工探索性测试 |

## CI/CD 集成策略

### 测试在流水线中的嵌入点

```
代码提交 → 静态检查 → 单元测试 → 构建 → 部署测试环境
                                              ↓
冒烟测试（P0自动化）→ 接口测试 → 功能测试 → 安全扫描
                                              ↓
性能测试（每晚）→ 预发布部署 → 全量回归 → 发布审批
```

### 质量门禁（Quality Gates）

| 阶段 | 门禁项 | 阈值 | 失败处理 |
|------|--------|------|---------|
| 构建阶段 | 单元测试覆盖率 | ≥ 80% | 构建失败 |
| 构建阶段 | 单元测试通过率 | 100% | 构建失败 |
| 部署后 | 冒烟测试通过率 | 100% | 阻断后续测试 |
| 测试阶段 | P0用例通过率 | 100% | 阻断发布 |
| 测试阶段 | P1用例通过率 | ≥ 95% | 需审批后发布 |
| 测试阶段 | 安全扫描高危漏洞 | 0个 | 阻断发布 |
| 测试阶段 | 性能基线偏离 | ≤ 110% | 需审批后发布 |

### 测试执行频率

| 测试类型 | 触发时机 | 执行时长目标 |
|---------|---------|------------|
| 单元测试 | 每次代码提交 | < 5分钟 |
| 接口测试 | 每次构建 | < 15分钟 |
| 冒烟测试 | 每次部署后 | < 10分钟 |
| 功能测试 | 每日/Sprint结束 | < 2小时 |
| 回归测试 | 发布前 | < 4小时 |
| 性能测试 | 每晚/Sprint结束 | 按场景设定 |
| 安全测试 | 每周/发布前 | 按工具设定 |
