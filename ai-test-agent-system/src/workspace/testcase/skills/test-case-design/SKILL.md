# 测试用例设计方法论 Skill

## 激活场景
- 需要为具体功能点生成测试用例
- 用户要求设计某模块或接口的测试用例
- 评估现有用例是否覆盖足够的设计维度

---

## 六大测试设计技术

### 技术一：等价类划分（Equivalence Partitioning）
**原理**：将输入域划分为若干等价类，每类中任意取一个值进行测试。

```
应用步骤：
1. 识别所有输入字段
2. 对每个字段划分有效等价类 + 无效等价类
3. 从每个等价类中取一个代表性值

示例 - 用户名字段（长度4-20个字符）：
┌─────────────────┬──────────────────┬─────────────────────┐
│ 等价类           │ 类别             │ 代表值               │
├─────────────────┼──────────────────┼─────────────────────┤
│ 长度4-20字符     │ ✅ 有效等价类     │ "testuser"（8位）    │
│ 长度1-3字符      │ ❌ 无效等价类     │ "abc"（3位）         │
│ 长度21+字符      │ ❌ 无效等价类     │ "a"×21（21位）       │
│ 含特殊字符       │ ❌ 无效等价类     │ "user@name"          │
│ 纯数字           │ ⚠️ 待定等价类    │ "12345678"           │
└─────────────────┴──────────────────┴─────────────────────┘
```

### 技术二：边界值分析（Boundary Value Analysis）
**原理**：在等价类边界处取值，错误往往出现在边界。

```
边界值选取规则（对于范围 [min, max]）：
  min-1  |  min  |  min+1  |  中间值  |  max-1  |  max  |  max+1

示例 - 年龄范围（18-65岁）：
┌──────┬─────────────────────────────────────────┐
│ 值   │ 用例说明                                  │
├──────┼─────────────────────────────────────────┤
│  17  │ 下边界-1（期望：拒绝，提示"年龄不符合要求"） │
│  18  │ 下边界（期望：接受）                        │
│  19  │ 下边界+1（期望：接受）                      │
│  41  │ 中间值（期望：接受）                        │
│  64  │ 上边界-1（期望：接受）                      │
│  65  │ 上边界（期望：接受）                        │
│  66  │ 上边界+1（期望：拒绝）                      │
│   0  │ 零值特殊处理（期望：拒绝）                   │
│  -1  │ 负数特殊处理（期望：拒绝）                   │
└──────┴─────────────────────────────────────────┘

字符串边界：空字符串("") / 单字符 / 最大长度 / 最大长度+1 / 超长字符串
数字边界：0 / 负数 / 最大整数 / 小数精度边界
日期边界：年份起止 / 月份28/29/30/31日 / 闰年 / 跨年
```

### 技术三：决策表法（Decision Table）
**原理**：用于多条件组合的业务规则测试，穷举条件组合。

```
示例 - 优惠券使用规则：
条件：
  C1: 订单金额 ≥ 100元
  C2: 用户是会员
  C3: 优惠券未过期

结果：
  A1: 可使用优惠券
  A2: 不可使用优惠券
  A3: 显示满减提示

┌────┬────┬────┬────┬────┬────┬────┬────┬────┐
│    │ 1  │ 2  │ 3  │ 4  │ 5  │ 6  │ 7  │ 8  │
├────┼────┼────┼────┼────┼────┼────┼────┼────┤
│ C1 │ T  │ T  │ T  │ T  │ F  │ F  │ F  │ F  │
│ C2 │ T  │ T  │ F  │ F  │ T  │ T  │ F  │ F  │
│ C3 │ T  │ F  │ T  │ F  │ T  │ F  │ T  │ F  │
├────┼────┼────┼────┼────┼────┼────┼────┼────┤
│ A1 │ ✅ │ ❌ │ ❌ │ ❌ │ ❌ │ ❌ │ ❌ │ ❌ │
│ A2 │    │ ✅ │ ✅ │ ✅ │ ✅ │ ✅ │ ✅ │ ✅ │
└────┴────┴────┴────┴────┴────┴────┴────┴────┘
→ 每列生成一个测试用例（可合并规则相同的列）
```

### 技术四：状态转换法（State Transition）
**原理**：针对对象状态机进行测试，验证状态转换的正确性。

```
应用对象：订单、用户账号、审批流程、工单状态

状态图示例 - 订单状态：
[待付款] --支付成功--> [待发货]
[待付款] --取消订单--> [已取消]
[待付款] --支付超时--> [已关闭]
[待发货] --商家发货--> [待收货]
[待发货] --取消申请--> [退款中]
[待收货] --确认收货--> [已完成]
[待收货] --申请退货--> [退款中]
[退款中] --退款成功--> [已退款]

测试用例设计：
1. 正向流程：待付款→待发货→待收货→已完成
2. 取消流程：待付款→已取消（取消按钮是否消失）
3. 超时流程：待付款→已关闭（超时触发）
4. 退款流程：待收货→退款中→已退款
5. 非法转换：已完成→不可再次申请退款（异常验证）
```

### 技术五：场景法（Scenario Method）
**原理**：模拟真实用户操作路径，串联多步骤的端到端场景。

```
场景构建公式：
  基本流（主成功场景）+ 备选流（分支/异常场景）

示例 - 电商下单场景：

基本流：
  选商品→加入购物车→选地址→选优惠券→提交订单→支付→完成

备选流清单：
  AF1: 商品库存不足（选商品后提示售罄）
  AF2: 地址为空（提交时提示添加地址）
  AF3: 优惠券不满足条件（不可选中灰显）
  AF4: 支付失败（跳转支付失败页，提供重试）
  AF5: 支付超时（订单关闭，返回商品列表）
  AF6: 网络中断（支付中断，订单状态处理）
  AF7: 重复提交（防重复下单校验）

每条备选流 = 1个完整测试用例
```

### 技术六：错误推测法（Error Guessing）
**原理**：基于经验和直觉推测最可能出现错误的场景。

```
高价值错误推测点：

【数据层】
- 空值：null / "" / " "（空白字符串） / undefined
- 特殊字符：' " < > & ; # % @ \ / | * ? 
- 超长输入：超过数据库字段长度限制
- 类型混淆：数字字段传字符串，日期格式错误
- 编码问题：中文、emoji、特殊Unicode字符

【业务层】
- 重复操作：快速双击提交按钮
- 并发操作：同时扣减库存/积分
- 跨会话操作：A用户操作B用户的数据
- 状态不一致：前端显示与后端状态不同步

【系统层】
- 文件上传：超大文件 / 0字节文件 / 可执行文件
- 图片处理：非图片文件改扩展名上传
- 时区问题：跨时区的时间计算
- 精度问题：金额计算的小数精度丢失
```

## 设计技术选择决策流程

面对具体功能点时，按以下流程选择最合适的设计技术：

```
Step 1: 识别功能特征
  ├─ 是否有明确的输入字段/参数？
  │    ├─ 是 → 进入 Step 2
  │    └─ 否 → 进入 Step 3
  └─ 是否有多个条件组合影响同一结果？
       ├─ 是 → 优先决策表法
       └─ 否 → 继续判断

Step 2: 输入特征判断
  ├─ 输入有明确取值范围？
  │    ├─ 是 → 等价类划分 + 边界值分析（组合使用）
  │    └─ 否 → 继续判断
  ├─ 输入是自由文本/无范围限制？
  │    ├─ 是 → 错误推测法（重点：特殊字符、超长、注入攻击）
  │    └─ 否 → 继续判断
  └─ 输入是枚举/下拉选项？
       ├─ 是 → 等价类划分（每个选项一个等价类）
       └─ 否 → 继续判断

Step 3: 业务特征判断
  ├─ 功能涉及对象状态变化？
  │    ├─ 是 → 状态转换法
  │    └─ 否 → 继续判断
  ├─ 功能是多步骤业务流程？
  │    ├─ 是 → 场景法（基本流 + 备选流）
  │    └─ 否 → 继续判断
  ├─ 功能有多个参数需组合验证？（参数 ≥ 3）
  │    ├─ 是 → 正交实验法/Pairwise（降低组合爆炸）
  │    └─ 否 → 继续判断
  └─ 历史有高发缺陷或经验可借鉴？
       ├─ 是 → 错误推测法
       └─ 否 → 基础功能验证用例
```

## 组合测试与 Pairwise 方法

当功能点涉及**3个及以上参数**且每个参数有**多个取值**时，全组合测试会产生爆炸性用例数。此时使用 Pairwise（两两组合）方法在覆盖率和效率间取得平衡。

### 适用场景

| 场景 | 示例 | 全组合数 | Pairwise数 |
|------|------|---------|-----------|
| 多字段表单 | 5个字段，每字段4个取值 | 1024 | ~20 |
| 接口参数 | 6个参数，每参数3个取值 | 729 | ~15 |
| 配置项验证 | 8个开关，每个开/关 | 256 | ~12 |

### Pairwise 设计原则

```
目标：确保任意两个参数的所有取值组合至少被覆盖一次。

示例：注册表单有3个参数
  用户名: [有效, 无效格式, 空]
  密码: [有效, 弱密码, 空]
  手机号: [有效, 无效, 空]

全组合：3×3×3 = 27 条
Pairwise：约 9-12 条（任意两个参数的两两组合均被覆盖）
```

> ⚠️ **重要**：Pairwise 不适用于安全测试和关键业务规则验证。对于安全相关参数（如密码强度、权限控制），必须执行全组合测试。

## 接口测试用例设计专项

### 接口测试设计框架

| 测试维度 | 设计要点 | 示例 |
|---------|---------|------|
| **契约验证** | 请求/响应字段名、类型、是否必填 | 字段 `user_id` 应为 `string` 类型且必填 |
| **状态码验证** | 各种场景下的 HTTP 状态码 | 成功→200，参数错误→400，未授权→401，服务端错误→500 |
| **参数边界** | 每个参数的有效/无效/边界值 | `page_size` 有效[1,100]，边界 0/1/100/101 |
| **参数组合** | 多参数同时传值/缺值/错值 | 同时缺失必填参数、部分参数格式错误 |
| **认证鉴权** | Token/Session/OAuth 的有效性 | 无效Token、过期Token、无Token |
| **幂等性** | 重复请求的处理 | 重复提交订单是否只生成一条记录 |
| **并发安全** | 并发调用同一接口 | 同时扣减库存是否超卖 |
| **数据一致性** | 接口操作后的数据状态 | 创建订单后数据库状态、缓存状态是否一致 |

### 接口测试用例模板

```markdown
### TC-[PROJECT]-API-[MODULE]-[NNN]：[接口名] - [场景描述]

| 字段 | 内容 |
|------|------|
| **用例编号** | TC-[PROJECT]-API-[MODULE]-[NNN] |
| **接口路径** | `METHOD /api/v1/xxx` |
| **用例标题** | [接口名] - [测试场景] |
| **优先级** | P0 / P1 / P2 |

**请求信息**：
```
Headers:
  Authorization: Bearer [token]
  Content-Type: application/json

Body:
  {
    "field1": "具体值",
    "field2": "具体值"
  }
```

**预期响应**：
```
Status: [HTTP状态码]

Body:
  {
    "code": [预期业务码],
    "message": "[预期消息]",
    "data": {
      "field1": "[预期值或类型]"
    }
  }
```

**数据层验证**：
- 数据库 [表名] 应 [新增/更新/删除] [具体记录描述]
- 缓存 [key] 应 [更新/失效]

**备注**：关联接口文档 [链接] | 关联需求 `REQ-XXX`
```

| Safari (最新) | - | - | ✅ | 核心功能 |

### 移动端兼容性矩阵

| 系统 | 小屏(375px) | 中屏(768px) | 大屏(1920px) | 测试重点 |
|------|------------|------------|-------------|---------|
| iOS (最新-2) | ✅ | ✅ | - | 手势、键盘适配 |
| Android (最新-2) | ✅ | ✅ | - | 返回键、权限弹窗 |

### 兼容性测试用例设计原则

1. **核心功能全覆盖**：所有 P0 用例必须在所有目标浏览器/设备上执行
2. **UI差异化处理**：不同分辨率下的布局适配、元素遮挡、滚动行为
3. **交互差异化处理**：触摸 vs 鼠标、物理返回键 vs 软件返回、系统权限弹窗
4. **降级策略验证**：老旧浏览器不支持新特性时的降级表现

## 设计质量红线

❌ **禁止出现**以下情况（如出现需立即修正）：
- 预期结果使用"正确显示"、"正常运行"、"成功"等无法客观验证的描述
- 一个用例包含多个不相关的验证点
- 步骤中使用"等待片刻"、"稍后"等无法量化的时间描述
- 前置条件不可重现（如"用户刚刚进行过某操作"）
- 缺少关键测试数据导致用例无法独立执行
- **接口测试用例缺少具体的请求 Body 和预期响应结构**
- **兼容性测试用例未明确标注目标浏览器/设备版本**
