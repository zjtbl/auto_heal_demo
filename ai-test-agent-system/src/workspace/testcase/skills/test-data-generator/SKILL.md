# 测试数据生成 Skill

## 激活场景
- 用例设计完成后需要填充测试数据
- 用户要求生成某类型字段的测试数据集

## 通用字段数据库

### 用户认证类数据

```yaml
手机号:
  有效数据:
    - "13812345678"    # 标准11位手机号
    - "15900000001"    # 最小有效号段
    - "19999999999"    # 最大有效号段
  边界数据:
    - "1381234567"     # 10位（下边界-1）
    - "138123456789"   # 12位（上边界+1）
    - "00000000000"    # 全零
    - "11111111111"    # 全相同数字
  无效数据:
    - ""               # 空字符串
    - " "              # 纯空格
    - "1381234567a"    # 含字母
    - "+8613812345678" # 国际格式
    - "abc"            # 纯字母
  安全数据:
    - "'; DROP TABLE users; --"   # SQL注入
    - "<script>alert(1)</script>" # XSS

密码:
  有效数据:
    - "Test@123456"    # 大小写+数字+特殊字符
    - "Aa1!Bb2@Cc3#"  # 复杂密码
  边界数据:
    - "Aa1!567"        # 最小长度（8位）
    - "Aa1!" + "x"×60 # 最大长度（64位）
    - "aa1!aaaa"       # 缺大写字母
    - "AA1!AAAA"       # 缺小写字母
    - "AaAaAa!!"       # 缺数字
    - "AaBb1234"       # 缺特殊字符
  无效数据:
    - "12345678"       # 纯数字
    - "password"       # 纯字母（常见弱密码）
    - "Aa1!567"[:-1]   # 7位（下边界-1）
    - ""               # 空密码

邮箱:
  有效数据:
    - "user@example.com"
    - "user.name+tag@domain.co.uk"
    - "user123@sub.domain.com"
  无效数据:
    - "userexample.com"    # 缺@
    - "user@"              # 缺域名
    - "@example.com"       # 缺用户名
    - "user @example.com"  # 含空格
    - "user@example"       # 缺顶级域
```

## 数据生成规则说明

生成测试数据时遵循以下优先级顺序：

```
P0（必须覆盖）：
  1. 最典型的有效值（Happy Path数据）
  2. 下边界值、上边界值
  3. 空值/null/零值

P1（应该覆盖）：
  4. 边界-1、边界+1
  5. 无效等价类代表值
  6. 中间值

P2（建议覆盖）：
  7. 特殊字符
  8. 超长字符串
  9. 类型错误（数字传字符串）

P3（有时间则覆盖）：
  10. SQL注入Payload
  11. XSS Payload
  12. Unicode边缘字符
```

## 数据隐私合规（Data Privacy Compliance）

企业级测试数据必须严格遵守数据保护法规（个保法、GDPR 等），**严禁使用未经处理的生产数据**。

### 脱敏规则矩阵

| 数据类型 | 生产数据示例 | 脱敏规则 | 测试数据示例 |
|---------|------------|---------|------------|
| 手机号 | 13812345678 | 保留前3位+后4位，中间替换为**** | 138****5678 |
| 身份证号 | 110101199001011234 | 保留前6位+后4位，中间替换为******** | 110101********1234 |
| 银行卡号 | 6222021234567890123 | 保留前6位+后4位，中间替换为**** | 622202****0123 |
| 邮箱 | zhangsan@company.com | 用户名部分保留首字符+****+末字符 | z****n@company.com |
| 姓名 | 张三 | 替换为随机姓氏+随机名 | 李四 |
| 地址 | 北京市海淀区xxx街道 | 保留省市，详细地址替换为"某街道某号" | 北京市海淀区某街道某号 |
| 密码 | （已加密存储） | 使用符合策略的合成密码 | Test@123456 |

### 合成数据生成策略

当需要大量逼真但非真实的测试数据时，使用以下策略：

**策略一：基于规则的合成**
```
手机号：使用虚拟号段 + 随机数字
  虚拟号段：170/171/165/167/162（虚拟运营商号段，不与真实用户冲突）
  示例：17012345678、16587654321

身份证号：基于真实规则构造，但使用未来日期或虚拟地址码
  地址码：使用测试专用地址码（如 999999 开头的虚拟编码）
  出生日期：使用 2000-01-01 等明显测试日期

邮箱：使用测试专用域名
  域名：@test.local、@qa.company.com、@example.com
```

**策略二：Faker 风格生成**
```
中文姓名：随机姓氏（李/王/张/刘等）+ 随机名
公司名称："测试企业" + 随机编号
地址：保留真实省市，详细地址使用"测试路X号"
```

**策略三：数据混淆（适用于从生产脱敏）**
```
步骤1：从生产环境导出数据样本
步骤2：应用脱敏规则处理敏感字段
步骤3：对关联字段进行一致性处理（如：脱敏后的身份证号需与出生日期一致）
步骤4：验证脱敏后数据不再可逆向还原
```

### 合规检查清单

```
□ 测试数据中不包含任何真实用户的个人信息
□ 敏感字段已按脱敏规则处理
□ 测试数据在传输和存储过程中加密
□ 测试环境访问权限已控制（最小权限原则）
□ 测试完成后数据已安全清理
□ 涉及第三方测试人员时，已签署保密协议
```

## 批量数据生成策略

当测试需要大量数据（性能测试、大数据量场景）时：

### 数据生成规模规划

| 测试类型 | 数据量级 | 生成策略 | 工具建议 |
|---------|---------|---------|---------|
| 功能测试 | 10-100条 | 手工构造 + 脚本辅助 | Python 脚本 |
| 接口测试 | 100-1000条 | 参数化脚本生成 | JMeter/Postman 数据文件 |
| 性能测试 | 1万-100万条 | 批量生成工具 | Python Faker、JMeter CSV |
| 大数据量测试 | 100万+条 | 数据库批量插入脚本 | SQL 批量插入、ETL 工具 |

### 批量数据一致性规则

生成大量数据时，必须确保数据间的业务一致性：

```
示例：生成 10万条订单数据

一致性约束：
1. 订单的 user_id 必须存在于用户表中
2. 订单的 product_id 必须存在于商品表中
3. 订单金额 = 商品单价 × 数量 - 优惠金额
4. 订单状态与支付记录状态一致（已支付订单必须有支付记录）
5. 订单创建时间 ≤ 支付时间 ≤ 发货时间 ≤ 完成时间

生成顺序：
Step 1: 生成用户数据（作为外键依赖）
Step 2: 生成商品数据（作为外键依赖）
Step 3: 生成订单主表数据（引用用户和商品ID）
Step 4: 生成订单明细表数据（引用订单ID和商品ID）
Step 5: 生成支付记录数据（引用订单ID，确保状态一致）
Step 6: 执行一致性校验脚本
```

| 环境 | 数据来源 | 数据生命周期 | 清理策略 |
|------|---------|------------|---------|
| 开发环境 | 开发人员本地构造 | 随开发迭代更新 | 每次迭代前重置 |
| 测试环境 | 测试团队统一管理 | 随测试周期更新 | 每次回归前重置基线数据 |
| 性能环境 | 性能测试专用数据集 | 长期保留基线数据 | 仅在基线变更时更新 |
| 预发布环境 | 生产脱敏数据 | 与生产同步周期 | 按生产同步周期更新 |

## 数据有效性验证

生成测试数据后，必须进行有效性验证：

### 验证规则

```
□ 格式验证：数据是否符合目标字段的格式要求
  示例：手机号是否为11位数字，邮箱是否包含@和域名

□ 边界验证：边界值是否确实处于边界
  示例：长度为10的字符串是否确实为10个字符（不是9也不是11）

□ 业务规则验证：数据是否满足业务约束
  示例：订单金额是否为正数，结束时间是否晚于开始时间

□ 关联一致性验证：关联数据之间是否逻辑一致
  示例：订单中的用户ID是否存在于用户表中

□ 唯一性验证：需要唯一的字段是否确实唯一
  示例：用户名、订单号、身份证号是否重复
```
