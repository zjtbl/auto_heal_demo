# 需求深度解析 Skill

## 激活场景
- 用户上传或粘贴需求文档（PDF、Word、Markdown、纯文本）
- 用户描述产品功能或业务流程
- 用户提供用户故事（User Story）或验收标准（AC）
- 用户要求分析某模块的测试范围

## 执行流程

### Step 1：文档结构解析
首先对输入文档进行结构化拆解，识别：
- **文档类型**：PRD / 接口文档 / 用户故事 / 设计稿描述 / 口头需求
- **功能模块列表**：按业务域划分（如：认证模块、订单模块、支付模块）
- **核心业务流程**：主流程 + 分支流程 + 异常流程
- **输入输出定义**：接口参数、UI字段、数据格式
- **业务规则**：约束条件、计算逻辑、状态机规则

### Step 2：测试点提取框架（INVEST原则）
对每个功能点，用以下维度提取测试要点：

| 维度 | 提取内容 | 示例 |
|------|---------|------|
| **功能性** | 核心功能是否按预期工作 | 登录成功跳转首页 |
| **数据** | 输入数据的类型、范围、格式 | 手机号11位数字 |
| **规则** | 业务约束和计算逻辑 | 优惠券不可叠加使用 |
| **状态** | 对象状态变化与转换 | 订单从待付款→已付款 |
| **交互** | 组件间联动与依赖关系 | 省市区三级联动 |
| **权限** | 不同角色的访问控制 | 管理员可删除，普通用户不可 |
| **异常** | 错误情况的处理策略 | 网络超时提示重试 |
| **性能** | 响应时间、并发、数据量 | 列表页加载不超过2秒 |

### Step 3：功能矩阵建立
输出标准功能矩阵表格：

```
## 功能测试矩阵

| 模块 | 功能点 | 测试要点 | 优先级 | 风险等级 | 测试类型 |
|------|--------|---------|--------|---------|---------|
| 用户认证 | 手机号登录 | 验证码有效期5min | P0 | 高 | 功能+安全 |
| 用户认证 | 第三方登录 | 微信/支付宝OAuth | P1 | 中 | 功能+兼容 |
```

### Step 4：风险识别与标注
重点标注以下高风险区域：
- 🔴 **安全风险**：认证、授权、数据加密、SQL注入点、XSS
- 🟠 **数据完整性风险**：事务处理、并发写入、数据一致性
- 🟡 **兼容性风险**：多端适配、浏览器差异、API版本兼容
- 🔵 **性能风险**：高频接口、大数据量操作、文件上传
- ⚪ **业务逻辑风险**：复杂计算、多条件判断、状态机

### Step 5：测试范围声明
生成测试范围说明：
```
## 测试范围声明

### 测试内容（In Scope）
- [列出所有纳入测试的功能模块]

### 排除内容（Out of Scope）
- [说明不在本次测试范围内的内容]

### 测试假设与前提
- [列出执行测试所需的环境、数据、权限假设]

### 依赖关系
- [列出与其他模块/系统的依赖]
```

## 输出规范

完成分析后，必须输出以下内容（按顺序）：

1. **📊 需求解析摘要**：文档类型、功能模块数量、核心业务流程数
2. **📋 功能测试矩阵**：完整的功能点×测试维度矩阵
3. **⚠️ 风险清单**：按风险等级排序的风险项
4. **📐 测试范围声明**：In/Out Scope 及假设条件
5. **📈 用例数量预估**：各模块预计测试用例数及总量

## 关键提示

> ⚡ **强制要求**：需求分析必须在生成任何测试用例之前完成。分析结果将直接影响测试策略选择和用例设计质量。
>
> 🎯 **深度要求**：不满足于表面功能描述，要挖掘隐含需求（如：登录成功后的安全日志记录）、负向需求（如：密码输入错误5次后锁定）、质量需求（如：响应时间要求）。
>
> 🔍 **疑问处理**：发现需求不明确、自相矛盾或有遗漏时，在分析报告中明确标注「需澄清问题」，并提出具体问题供用户确认。

## 与 RAG 检索结果的衔接规范

本 Skill 的输入包含两类信息：
1. **用户原始需求**（PRD/描述/图片等）
2. **`rag-query` Skill 的检索报告**（历史用例、业务规则、缺陷模式等）

### 检索结果的消纳规则

| 检索结果类型 | 如何纳入需求分析 |
|-------------|----------------|
| 业务规则与约束 | 直接纳入功能矩阵的「测试要点」列，标注来源 `[RAG·高置信]` |
| 历史测试用例 | 用于验证当前需求的功能点是否完整（对比历史覆盖范围） |
| 历史缺陷模式 | 纳入风险清单，标注为「已知高频缺陷区域」 |
| 实体定义 | 统一术语使用，确保分析报告中术语与知识库一致 |
| 关联实体/流程 | 补充隐含需求（如：用户注册需考虑与权限模块的联动） |
| 检索缺口 | 在「测试假设」中明确标注「基于需求原文，未检索到历史参考」 |

### 检索结果与需求冲突的处理

当 RAG 检索到的历史规则与当前需求描述冲突时：
1. 在分析报告中标注「⚠️ 知识库冲突」
2. 列出冲突双方：知识库规则 vs 当前需求描述
3. 优先采信**当前需求描述**（需求是最新权威），但建议用户确认
4. 若需求描述不明确，采信**高置信度的知识库规则**作为假设基础

## 非功能需求分析框架（NFR Analysis）

企业级测试必须覆盖功能性需求（FR）和非功能性需求（NFR）。在解析功能需求的同时，系统性识别以下 NFR 维度：

### 性能需求（Performance）

| 检查项 | 需求描述特征 | 测试关注点 |
|-------|------------|-----------|
| 响应时间 | "X秒内返回" / "不超过Y毫秒" | 接口级基准测试、前端渲染时间 |
| 吞吐量 | "支持X并发" / "每秒Y笔交易" | 负载测试、压力测试 |
| 资源占用 | "内存不超过X" / "CPU占用低于Y%" | 性能监控、资源泄漏检测 |
| 数据量 | "支持X万条数据" / "Y GB存储" | 大数据量场景测试、分页/分库验证 |

> 若需求未明确性能指标，在「测试假设」中标注「性能需求未量化，建议补充 SLA 指标」，并基于行业常见标准设计基线用例。

### 安全需求（Security）

| 安全域 | 检查项 | 需求描述特征 |
|-------|--------|------------|
| 认证 | 身份验证强度、多因素认证、会话管理 | "需要登录"、"支持指纹/人脸" |
| 授权 | 角色权限、数据隔离、越权防护 | "管理员可操作"、"仅查看自己的数据" |
| 输入安全 | SQL注入、XSS、CSRF、文件上传安全 | 任何涉及用户输入的功能 |
| 数据保护 | 加密传输、存储加密、敏感信息脱敏 | "密码加密"、"身份证号脱敏" |
| 审计 | 操作日志、访问记录、可追溯性 | "记录操作日志"、"支持审计查询" |

> 若需求未明确安全要求，默认执行**基础安全测试**（OWASP Top 10 核心项），并在风险清单中标注「安全需求未明确，仅覆盖基础项」。

### 兼容性需求（Compatibility）

| 维度 | 检查项 | 测试策略 |
|------|--------|---------|
| 浏览器 | Chrome/Firefox/Safari/Edge 及版本 | 矩阵覆盖主流版本 |
| 设备 | 手机/平板/PC 分辨率适配 | 响应式布局验证 |
| OS | iOS/Android/Windows/macOS 版本 | 核心功能跨平台验证 |
| 数据兼容 | 历史数据迁移、新旧版本数据互操作 | 数据升级/降级测试 |
| API兼容 | 接口版本兼容、字段废弃策略 | 契约测试、回归验证 |

### 可用性与可靠性需求

| 维度 | 检查项 | 测试关注点 |
|------|--------|-----------|
| 可用性 | 错误提示清晰度、操作反馈即时性、无障碍支持 | 错误场景用户体验 |
| 可靠性 | 容错恢复、故障转移、数据一致性保障 | 异常注入测试、混沌测试 |
| 可维护性 | 日志完整性、监控埋点、告警策略 | 运维场景验证 |

## 需求质量评估（PRD Completeness Check）

对输入的需求文档进行质量评分，识别缺失项：

### 完整性检查清单

```
□ 功能描述：每个功能点有明确的输入、处理逻辑、输出
□ 业务规则：约束条件、计算公式、状态流转规则已定义
□ 异常处理：错误场景、边界条件、异常流程已说明
□ 数据定义：字段类型、长度、取值范围、格式要求已定义
□ 权限定义：不同角色的操作权限已明确
□ 性能要求：响应时间、并发数、数据量等已量化
□ 安全要求：认证、授权、数据保护等已明确
□ 兼容性要求：支持的浏览器、设备、OS版本已列出
□ 接口定义：API 的请求/响应格式、错误码已定义
□ 依赖关系：与其他模块/系统的依赖已说明
```

### 评分标准

| 得分 | 等级 | 说明 |
|------|------|------|
| 10/10 | 优秀 | 需求完整，可直接进入测试设计 |
| 7-9/10 | 良好 | 少量缺失，可在测试设计阶段补充假设 |
| 4-6/10 | 一般 | 明显缺失，需标注「需澄清问题」 |
| 0-3/10 | 差 | 严重不足，建议暂停测试设计，先补充需求 |

## 隐性需求挖掘方法（Implicit Requirements Mining）

企业级需求分析不能只依赖文档上写的内容，必须系统性挖掘隐性需求。

### 负向需求挖掘（Negative Requirements）

对每个功能点，主动思考"什么情况下不应该发生"：

| 功能点 | 正向需求 | 负向需求（隐性） |
|-------|---------|----------------|
| 用户登录 | 正确密码可登录 | 密码错误不应登录；密码错误5次应锁定；锁定期间不应允许登录 |
| 订单提交 | 库存充足可下单 | 库存不足不应下单；未登录不应下单；商品已下架不应下单 |
| 数据删除 | 管理员可删除 | 非管理员不应删除；已关联其他数据的不应直接删除；删除后不应影响历史记录 |

### 质量属性需求挖掘

对每个功能点，检查是否隐含以下质量属性要求：

```
□ 一致性：同一操作在不同入口的结果是否一致
□ 原子性：复合操作是否支持回滚（如：支付失败时订单状态是否回退）
□ 幂等性：重复执行同一操作是否产生副作用（如：重复提交订单）
□ 并发安全：多用户同时操作同一资源时的行为
□ 时序敏感：操作顺序是否影响结果（如：先改密码再登录 vs 先登录再改密码）
□ 数据完整性：操作中断（如网络断开）时数据是否一致
```

### 合规性需求挖掘

根据业务领域，自动检查是否需要满足以下合规要求：

| 行业 | 合规标准 | 测试关注点 |
|------|---------|-----------|
| 金融 | 等保2.0、PCI-DSS、JR/T 0071 | 数据加密、访问控制、审计日志、交易一致性 |
| 医疗 | HIPAA、等保2.0、GB/T 35273 | 患者隐私保护、数据脱敏、操作可追溯 |
| 政务 | 等保2.0、GB/T 25000.51 | 国产化适配、数据主权、安全审计 |
| 电商 | 个人信息保护法、GDPR（如有海外用户） | 用户同意管理、数据删除权、跨境传输 |
| 教育 | 个人信息保护法、未成年人保护 | 年龄验证、家长同意、内容过滤 |

## 需求变更影响分析

当用户提供的不是全新需求而是需求变更时，执行以下影响分析：

### 变更影响矩阵

| 变更类型 | 影响范围 | 测试策略调整 |
|---------|---------|------------|
| 新增功能 | 新增模块 + 关联模块 | 全面设计 + 关联模块回归 |
| 功能修改 | 当前模块 + 上下游模块 | 聚焦变更点 + 端到端场景回归 |
| 规则调整 | 所有依赖该规则的模块 | 决策表全量重检 + 规则相关用例回归 |
| 接口变更 | 所有调用方 | 契约测试 + 集成测试 |
| 性能优化 | 全链路 | 基准测试对比 + 压力测试 |
| 安全加固 | 全系统 | 安全专项测试 + 渗透测试 |

### 变更分析输出模板

```markdown
## 需求变更影响分析

### 变更概述
- 变更内容：[简述]
- 变更类型：[新增/修改/删除/优化]
- 影响模块：[列表]

### 影响分析
| 模块 | 影响程度 | 影响说明 | 测试行动 |
|------|---------|---------|---------|
| [模块A] | 高 | [说明] | [行动] |
| [模块B] | 中 | [说明] | [行动] |

### 回归范围建议
- **必须回归**：P0 用例 + 变更直接影响的模块用例
- **建议回归**：P1 用例 + 上下游关联模块用例
- **可选回归**：P2/P3 用例（基于风险评估）
```
